01Executive summary

This plan delivers the smallest viable set of IT services to run a secure, multi-site network that never touches the public internet. It covers user accounts, authentication, authorization, logging, and auditing — and nothing that isn't required to operate them safely. Every component is modern, open-source, and chosen for being lightweight enough to run on commodity hardware while still scaling past 250 endpoints. This minimal set is the core platform — capability feature packs (collaboration, communications, developer services) bolt on later without re-architecting it.

Services are organized into eight layers, built strictly bottom-up: each layer depends only on the ones below it, so the network is provably correct at every step. A clear distinction is drawn between the always-connected core (the network and its services) and an optional offline endpoint image — a hardened, self-contained build that operates on a laptop with no network connection at all, then reconciles its accounts and audit trail when it next rejoins.

The build is delivered in tiers. Tier 0 (core) is the eight-layer foundation that runs accounts, auth, logging and audit. Tier 1+ (feature packs) add a wiki, webmail, file sharing, and VoIP/VTC as self-contained services on the same platform — adopt only what each site needs. A lightweight, internet-connected demo/dev twin (5–15 users, Docker Compose) mirrors the production tools for evaluation and development; its files are in Appendix B.

02Design principles & constraints

P1 · AIR-GAPPED BY DEFAULT

No internet egress, ever. All software, OS packages, and container images enter through a controlled offline mirror. Nothing assumes a public endpoint (no public NTP, no public CA, no SaaS).

P2 · MINIMAL SERVICE SET

Only what's needed to run accounts, auth, authz, logging, and auditing — plus the foundation those require. No optional service ships without a named owner and a reason.

P3 · LIGHTWEIGHT & MODERN

Single-binary or single-container tools preferred (K3s over full K8s, lldap over OpenLDAP, step-ca over EJBCA). Declarative, reproducible, low operational burden.

P4 · STRICTLY LAYERED

Each layer depends only on layers beneath it. Build, test, and harden bottom-up; never stand up a service before its dependencies exist and are verified.

P5 · PARTIAL OFFLINE, NOT FULL

The core network is always-on within the air-gap. "Offline" applies to a subset: a self-contained endpoint image that runs disconnected and reconciles on return.

P6 · AUDIT EVERYTHING

Every authentication, authorization decision, and administrative action is logged centrally and made tamper-evident. Audit is a first-class layer, not an afterthought.

03Architecture overview

The original five-layer brief is preserved and refined into eight: the hardware layer is split into facility and compute fabric; source control & configuration is promoted to its own layer (Gitea, the registry, and GitOps) so the system has a single, auditable source of truth; and a dedicated Observability & Audit layer designs logging and auditing in rather than bolting them on. Two concerns — Backup & DR and Security & Hardening — cut across all eight.

Mapping to the original brief: Layer 1 (HW) → L1+L2 · Layer 2 (DHCP/DNS/NTP) → L3 · Layer 3 (Servers) → L4 · Layer 4 (Auth) → L6 · Layer 5 (Endpoints) → L8. New: L5 Supply Chain & Config, L7 Observability & Audit.

04Layer-by-layer detail

L1Facility & physical plant

The ground truth: secured server rooms at each site, conditioned power (dual-feed UPS + generator where available), cooling, and structured cabling. Physical access is controlled and logged — the air-gap is only as strong as the door. Sites are linked over a private point-to-point fiber (no carrier internet); fully isolated "island" sites that reconcile by escorted media transfer are supported as a fallback.

Locked racks + access logDual UPS / PDUPrivate inter-site fiberNo external WAN uplink

L2Hardware & storage fabric

Commodity x86 servers in a hyper-converged arrangement: each primary site runs a 3-node cluster (CPU + RAM + NVMe) providing both compute and replicated storage via Ceph — three nodes give quorum and survive a single-node loss. Branch sites run 1–2 nodes. Networking is redundant managed L3 switches with VLAN support plus a firewall/router enforcing segmentation (with no internet egress path to enable).

RolePrimary siteBranch site
Compute / storage nodes3 × HCI (Ceph)1–2 × node
Backup target1 × Proxmox Backup Serverreplicates to primary
Network2 × L3 switch, 1 × firewall1 × L3 switch

L3Network foundation services

The services everything else assumes. DHCP + DNS run on a redundant pair (Technitium, or dnsmasq for the simplest footprint) — internal zones only, no forwarders. NTP is the subtle one: with no public time source, one appliance per site is a GPS-disciplined stratum-1 clock feeding chrony on every host. Accurate time is non-negotiable — TLS, Kerberos, tokens, and audit ordering all break when clocks drift.

An internal PKI (step-ca) issues every TLS certificate from a site-local root, replacing the public CAs that don't exist here — it is the trust anchor for the mutual-TLS used everywhere above. Keep this layer ruthlessly small and boring: it is the foundation the whole platform assumes is always correct. (The offline mirror, container registry, and source control sit one layer up, in L5.)

L4Compute platform

Two tools, one clear split — not three ways to do the same thing. Proxmox VE is the virtualization substrate: it clusters the physical nodes and runs VMs and LXC containers. K3s is the single orchestrator for production services — a 3-node cluster with an embedded HA datastore that self-heals and scales without full Kubernetes overhead. Everything containerized runs on K3s; Proxmox carries only what must be a VM (the PKI root, the identity database) plus the K3s nodes themselves. Docker Compose is deliberately not a third production platform: it is reserved for two narrow jobs — bootstrapping the registry/mirror before K3s exists, and the lightweight demo/dev twin (§05). Configuration is declarative: Ansible provisions hosts, and a GitOps controller (Flux) reconciles K3s from Git (L5).

L5Software supply chain & configuration control

The system's single source of truth and the only seam where the outside world enters. An offline package mirror (OS packages, PyPI, OS images) and an OCI container registry (Harbor for scanning + RBAC, or Zot for minimalism) hold every artifact the air-gap runs — nothing is ever pulled from the internet. Gitea hosts all source code and Infrastructure-as-Code: the declarative definition of the entire environment, promoted to its own layer because everything above is built from it.

Updates are controlled, never ad-hoc. External artifacts enter only through a reviewed offline-ingestion process into the mirror/registry; from there a change is just a Git commit — reviewed, CI-built and signed, pushed to the registry, then reconciled onto K3s by Flux (GitOps). Every change is versioned, attributable, and reversible. The update flow is detailed in §06.

L6Identity, authentication & authorization

The single source of truth for who and what they may do. lldap is the lightweight directory of users and groups (single binary, web UI). Keycloak federates it and acts as the identity provider, issuing OIDC/SAML single sign-on to every web service (Guacamole, Harbor, Gitea, Grafana, Proxmox). Linux hosts authenticate via SSSD against the same directory. MFA is enforced with TOTP and WebAuthn/hardware keys — both work fully offline, unlike SMS. Authorization is role-based: groups in lldap map to Keycloak roles map to per-service permissions. Optional FreeRADIUS adds 802.1X port-level network access control.

See FIG 4 for the end-to-end authentication flow.

L7Observability & audit

Logging and auditing as a designed layer. Grafana Loki aggregates logs (shipped by lightweight Alloy/Promtail agents); Prometheus + Grafana cover application and service metrics and dashboards, while Zabbix watches infrastructure availability — device up/down, hardware health, and SNMP from switches, firewalls, and UPS units (a Prometheus SNMP-exporter is a fine single-stack alternative). The audit backbone is Wazuh — a single open-source platform giving host intrusion detection, file-integrity monitoring, log analysis, and CIS compliance scoring. Host auditd events, Keycloak login/admin events, and Guacamole session records all flow to it. Audit data lands in append-only, restricted-access storage with periodic integrity hashing, so the record is tamper-evident. Alerting routes to dashboards and internal SMTP.

Loki + Alloy (logs)Prometheus + GrafanaZabbix (infra / SNMP)Wazuh (audit/SIEM/FIM)auditd on all hostsappend-only audit store

L8Access & endpoints

Apache Guacamole is the clientless access gateway: users reach RDP/VNC/SSH desktops and apps from a browser, with SSO via Keycloak and every session recordable for audit. Most users sit at stateless zero/thin clients that boot straight into a locked-down browser pointed at Guacamole — no local data, nothing to steal. The compute for those desktops lives on the platform (L4). Separately, an optional offline image serves users who must work on a laptop with no network at all; it is detailed below and in FIG 5.

The optional offline endpoint image

For users who must work on a laptop with no connection to the network at all, a hardened, immutable OS image (e.g. an atomic Linux build) runs entirely self-contained. It carries the apps and the slice of data the role needs, authenticates locally against a cached credential, and writes its own signed audit log. It is deliberately not the whole system — it's a bounded, time-boxed extension of it.

CONTROLS
  • Full-disk encryption (LUKS) + verified boot
  • Cached credential with a hard expiry (e.g. 30 days)
  • Local auditd log, hash-chained & signed
  • Lockout once the offline window lapses
ON RECONNECT
  • Identity + image-integrity re-checked
  • Local audit log uploaded to Wazuh
  • Credentials & policy refreshed; clock re-synced
  • Remote-wipe honored if device reported lost

05Environments: dev, test, production & demo

One definition, several environments. The same Infrastructure-as-Code in Gitea (L5) deploys to isolated environments so every change is proven before it reaches users. Dev and Test/Staging are separate K3s namespaces (or a small dedicated cluster) fed by the same GitOps pipeline; Production is the air-gapped multi-site platform; and a demo/dev twin runs on a single internet-connected cloud VM under Docker Compose for evaluation, training, and external development.

06Configuration control & updates

Nobody configures production by hand. The desired state of the platform lives in Gitea as Infrastructure-as-Code, and every change — a settings tweak or a new app version — follows the same reviewed path. That makes changes versioned, attributable, and reversible: a rollback is a git revert, and the prior signed artifact is still in the registry.

07Feature layers (optional expansion)

Beyond the core, capability packs add user-facing services. Each runs on the same platform (K3s in production, Compose in the demo), signs users in through Keycloak SSO, and is logged and audited like everything else. They are modular — adopt only what a site needs, in any order.

08Build order & phasing

Build strictly bottom-up. Each phase delivers a working, hardened layer that the next depends on — there is no point standing up identity before the PKI and time source it trusts exist. The plan below assumes a ~9-month primary-site build with branch rollout at the end; phases overlap only where there is no hard dependency.

09Backup & disaster recovery

A 3-2-1 strategy lived entirely inside the air-gap. Proxmox Backup Server takes deduplicated, incremental backups of every VM and LXC; application databases (Keycloak, Gitea, Harbor, Guacamole) also get logical dumps. Backups replicate over the private link to a second site, and a rotating set of offline media is held in a separate secured location. Restores are runbook-driven and tested on a schedule — an untested backup is a guess.

Special cases: the PKI root key is kept offline in a safe (HSM or sealed media); the append-only audit store is replicated but never overwritten. RPO/RTO targets are starting points — confirm against business impact per service.

ansible/site.ymlProvisioning

Installs Docker + Compose and brings the stack up

# =============================================================================
#  Provision the demo cloud server and bring up the stack.
#  Usage:
#     ansible-galaxy collection install community.docker
#     ansible-playbook -i inventory.ini site.yml
#  Expects the demo/ files (docker-compose.yml, Caddyfile, .env, init-db.sh,
#  prometheus.yml, realm-export.json, compose.comms.yml) in ../demo relative
#  to this playbook, and a populated .env (copy from .env.example first).
# =============================================================================
- name: Deploy Offline Network System — demo stack
  hosts: demo
  become: true
  vars:
    app_dir: /opt/ons-demo
    files:
      - docker-compose.yml
      - compose.comms.yml
      - Caddyfile
      - init-db.sh
      - prometheus.yml
      - realm-export.json
      - .env

  tasks:
    - name: Install prerequisites
      ansible.builtin.apt:
        name: [ca-certificates, curl, gnupg]
        update_cache: true

    - name: Add Docker apt key
      ansible.builtin.shell: |
        install -m 0755 -d /etc/apt/keyrings
        curl -fsSL https://download.docker.com/linux/ubuntu/gpg \
          | gpg --dearmor -o /etc/apt/keyrings/docker.gpg
        chmod a+r /etc/apt/keyrings/docker.gpg
      args:
        creates: /etc/apt/keyrings/docker.gpg

    - name: Add Docker apt repository
      ansible.builtin.apt_repository:
        repo: "deb [arch=amd64 signed-by=/etc/apt/keyrings/docker.gpg] https://download.docker.com/linux/ubuntu {{ ansible_distribution_release }} stable"
        filename: docker

    - name: Install Docker Engine + Compose plugin
      ansible.builtin.apt:
        name: [docker-ce, docker-ce-cli, containerd.io, docker-compose-plugin]
        update_cache: true

    - name: Ensure application directory
      ansible.builtin.file:
        path: "{{ app_dir }}"
        state: directory
        mode: "0750"

    - name: Copy stack files
      ansible.builtin.copy:
        src: "../demo/{{ item }}"
        dest: "{{ app_dir }}/{{ item }}"
        mode: "0640"
      loop: "{{ files }}"

    - name: Make init-db.sh executable
      ansible.builtin.file:
        path: "{{ app_dir }}/init-db.sh"
        mode: "0750"

    - name: Bring up the stack
      community.docker.docker_compose_v2:
        project_src: "{{ app_dir }}"
        files: [docker-compose.yml]
        state: present
        pull: always
